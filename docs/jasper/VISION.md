# Jasper Vision

## One Line

Jasper doubles the operator.

If Reif can do `X` alone, Reif with Jasper should be able to do `2X` because Jasper exists.

Jasper is not the language model itself. The language model is a core substrate.
The product goal is to build a co-founder for life: a persistent brain in a box that can think, remember, orient, act, and stay with the operator across years.

That doubling should show up in:

- clarity
- memory
- execution speed
- consistency
- protection

## Why The Vision Must Stay Simple

A bold product vision is useful because it creates orientation.

Most engineering progress does not come from a few giant decisions. It comes from hundreds of small tradeoffs made during normal work:

- what to build first
- what to defer
- which implementation path to choose
- whether to optimize polish or leverage

The vision should act like a magnetic field for those choices.

## Product Heuristic

When there are two or three reasonable options, choose the one that gets Jasper to a reliable `2X` force multiplier fastest.

That means Jasper should generally prefer work that:

- reduces user setup and operating friction
- increases day-to-day leverage
- turns memory into useful action
- moves Jasper from reactive chat to proactive operation
- preserves consent boundaries while hiding infrastructure complexity

## What `2X` Means For Jasper

### 1. `2X` Clarity

Jasper should reduce confusion, forgotten context, and repeated explanation.

Signs this is working:

- the user re-explains less
- active projects stay coherent across sessions
- Jasper can summarize what matters now

### 2. `2X` Memory

Jasper should remember what happened, what matters, and what was decided.

Signs this is working:

- past decisions are retrievable
- commitments are not silently lost
- people, preferences, and constraints stay available

### 3. `2X` Execution

Jasper should compress the time between intent and action.

Signs this is working:

- research turns into decisions faster
- repeated workflows become reusable tools or routines
- Jasper can prepare, draft, route, and execute multi-step work

### 4. `2X` Protection

Jasper should help the household notice risk earlier and respond faster.

Signs this is working:

- anomalies are surfaced quickly
- important systems are monitored continuously
- escalation happens with useful context instead of noise

### 5. `2X` Continuity

Jasper should keep work moving between sessions instead of resetting to zero.

Signs this is working:

- the next useful action is obvious
- Jasper can resume long-running threads
- background monitoring and queued work continue while the user sleeps

## What Jasper Is Not

Jasper should not become:

- a generic assistant with no household memory
- just a language model in a chat window
- a bag of disconnected features
- a product that leaks provider plumbing into normal conversation
- a system that depends on constant micromanagement to stay useful

## Co-Founder Standard

Jasper should be built to the standard of a lifelong co-founder, not a disposable assistant.

That means:

- Jasper should accumulate context instead of resetting to zero
- Jasper should improve judgment and execution, not just produce fluent text
- Jasper should be able to carry priorities, constraints, and commitments over long periods
- Jasper should become more useful with time, not less
- Jasper should feel like a real operating intelligence, a brain in a box, rather than a thin wrapper around a model API

## North-Star Test

Jasper is moving in the right direction if a normal day feels like this:

- fewer dropped threads
- fewer repeated explanations
- faster movement from question to action
- better daily awareness of household state
- more confidence that important work will not be forgotten

If a proposed change does not materially improve leverage, memory, execution, protection, or continuity, it is probably not the next priority.
